description: Integrate queued Workforest branches into local main with conservative verification. Use when processing queued branches into local main.

Use when processing queued branches into local main.

Core requirements: before listing or inspecting queued entries, acquire the integration lock with .pi/scripts/integration.mjs acquire-lock and keep it held for the entire integration run. Store the returned token and always release it with .pi/scripts/integration.mjs release-lock --token <token> once integration is finished, fails, or is interrupted. Use acquire-lock --force only after verifying the previous integration process died and left workforest-main.lock unreleased. Process refs one at a time from refs/workforest/integration-ready/<timestamp>/<branch>; skip or refresh stale entries before integration; verify queued SHA still equals branch HEAD; compare incoming changes against merge base and current main; keep history linear where possible; resolve conflicts narrowly without discarding unrelated local main changes; leave local main unchanged if merge/check verification fails. Use one isolated wf task worktree per queued entry, created with wf task new and removed with wf task delete; never use raw git worktree for integration worktrees or rebase a queued branch in its existing worktree. Run pnpm check before each main fast-forward: once in the integration task worktree, then again while holding workforest-main.lock immediately before fast-forwarding local main. Hold the same lock while rebasing helper onto latest main, checking, fast-forwarding designated main worktree, and pushing origin main:main. After all queued changes are merged into local main, always run pnpm build in the designated main/ worktree before final response; build failure is a blocker. If queue is empty at start, report nothing to integrate and stop without pnpm check or pnpm build.

After each successful push, run .pi/scripts/integration.mjs sync-worktree <branch|id> --target <pushed-main-sha> for the original integrated branch worktree. Pass the exact checked, fast-forwarded, pushed commit, especially after conflicts/manual integration, so integration-time adjustments backport and wf delete ancestry proof survives. The helper must skip dirty, stale, or missing source worktrees; report skipped reasons. After at least one original source worktree was integrated or synced, final cleanup offer is mandatory: before final response, list each eligible original worktree and ask whether to run wf delete. Run wf delete only after user confirmation and only for worktrees whose integration is verified in Git history. Syncing source worktrees, deleting ephemeral integration tasks/helper branches, running final main/ build, or reporting empty queue does not satisfy this offer. Do not offer original worktree cleanup while queued integrations remain.

Auditor: for non-trivial/risky/broad diffs, run the project-scoped integration-auditor subagent read-only before editing. Invoke it with the `subagent` tool using `agentScope: "both"` so the agent in `.pi/agents/integration-auditor.md` loads; give it the branch, merge base, and queued SHA to audit. Never use mini/nano-class models for integration audits. The auditor runs in an isolated context and its tool calls stream to you as it works; treat its final message as a self-contained verdict (SAFE, RISKY with exact file:line areas, or BLOCK with reason). If no project subagent is available, fall back to your own read-only diff review with a high-capability model (opus-class or gpt-5.5-class); do not gate integration on a mini/nano-class review.

Workflow: acquire the run lock first, then install a shell trap so release-lock --token <token> runs on normal exit, error, or interruption. One safe shape is: lock_json="$(.pi/scripts/integration.mjs acquire-lock)"; lock_token="$(node -e 'process.stdin.on("data", d => console.log(JSON.parse(d).token))' <<<"$lock_json")"; trap '.pi/scripts/integration.mjs release-lock --token "$lock_token"' EXIT. If acquire-lock reports an existing lock, inspect it and active processes before retrying with acquire-lock --force. After the lock is held, list queue with .pi/scripts/integration.mjs list. Pick oldest ready entry not stale or already integrated. Inspect diff and run auditor when required. Create task worktree using the procedure below. Cherry-pick queued commits in order, or only unapplied commits if equivalent changes are already on main under different SHAs. Run pnpm check in the task worktree. While still holding workforest-main.lock, rebase helper onto latest local main, rerun pnpm check, fast-forward designated main worktree, and push origin main:main. Capture pushed main SHA, sync original source worktree, dequeue integrated entry, remove ephemeral task worktree, delete merged helper branch. Repeat until no ready entries remain. After the last queued entry lands on local main, run pnpm build from main/. Closeout gate: once all queued integrations are complete, main/ build passes, temporary tasks/helper branches are cleaned up, and the release trap has run or is guaranteed to run, offer wf delete for eligible original worktrees or explicitly say none are eligible.

Integration task worktrees: use task name integrate-<queue-id>-<branch-slug>; Workforest creates the branch with the configured tomdale/ prefix. From the clean parent main worktree for that repo, run wf task new integrate-<queue-id>-<branch-slug> (add --repo <repo> when running from a workspace), cd to the task worktree, git merge --ff-only main, git cherry-pick <merge-base>..<queued-sha>, pnpm check. Before creation, verify the task name is absent from wf task list and no helper branch/path collision exists. wf task new branches from parent repository committed HEAD; git merge --ff-only main verifies the helper task is based on current integration base before queued commits are applied. After push/dequeue, remove ephemeral task with wf task delete integrate-<queue-id>-<branch-slug> (add --repo <repo> if needed). Keep failed integration task worktrees for diagnosis until successful integration or deliberate abandonment. wf delete is only for original integrated worktrees after user confirmation.

User-facing output: concise CLI status, not narrative postmortem. Use one short heading, two-space indentation, stable labels, symbols: ◆ active, ✔︎ completed, ▲ warning/skipped/stale/non-blocking auditor issue, ✗ failed/blocker, ● info. Keep command output summaries separate from commentary. Progress example: Integration; ◆ Inspecting tomdale/example; ● Queue: refs/workforest/integration-ready/...; ● Base: 34d5f66; ● Head: 123abcd. For each queued entry report when known: Queue ref, Branch, Head SHA, Merge base, Status integrating/already integrated/stale/skipped/failed/integrated, Verification command/result, Push result, Source worktree sync result. Final success shape: Integration complete; ✔︎ Queue: empty; ✔︎ main: <sha>; ✔︎ origin/main: <sha>; ✔︎ Verification: pnpm check; ✔︎ Build: pnpm build in main/; ✔︎ Pushed: origin main:main; Integrated worktrees with ● <name-or-path>; Cleanup with ◆ Run wf delete for these integrated worktrees? If none eligible: Cleanup; ● No eligible original worktrees. Put warnings, skipped sync reasons, failed pushes, dirty source worktrees, and failed main/ builds under Warnings with ▲ lines, not buried in prose.
